## 数据库设计

### 1. 数据库职责划分

本系统使用 MySQL、Milvus、MongoDB 和 MinIO 分别承担不同类型的数据存储职责。

```text
MySQL：
保存常规关系型数据，包括企业配置、业务线配置、业务线与知识库绑定关系、知识库、文档、图片元数据、模型配置等。

Milvus：
保存知识库 chunk、向量、主体识别结果、图片 URL 等检索相关数据。

MongoDB：
保存聊天 session、聊天消息、问题改写结果、引用快照等相对非结构化数据。

MinIO：
保存原始上传文档和文档中解析出来的图片。
```

整体关系：

```text
MySQL business_line_config
        ↓ business_line_id
MySQL business_line_knowledge_base
        ↓ kb_id
MySQL knowledge_base
        ↓ kb_id
MySQL document
        ↓ doc_id
MinIO 原始文件 / 图片
        ↓ image_url
Milvus kb_chunks
        ↓ references
MongoDB chat_messages
```

### 2. MySQL 表设计

------

#### 2.1 system_config

用途：

保存企业基础配置。第一版默认只有一条配置记录。

表名：

```text
system_config
```

建表语句：

```sql
CREATE TABLE system_config (
    id VARCHAR(64) PRIMARY KEY,
    company_name VARCHAR(255) NOT NULL,
    industry VARCHAR(255),
    company_description TEXT,
    business_scope TEXT,
    answer_tone VARCHAR(255),
    welcome_message TEXT,
    fallback_message TEXT,
    disclaimer TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

默认记录：

```text
id = "default"
```

------

#### 2.2 business_line_config

用途：

保存业务线配置。一个公司可以有多个业务线，每个业务线拥有独立的聊天框嵌入代码，并通过绑定关系关联一个或多个知识库。

表名：

```text
business_line_config
```

建表语句：

```sql
CREATE TABLE business_line_config (
    id VARCHAR(64) PRIMARY KEY,
    business_line_name VARCHAR(255) NOT NULL,
    business_line_description TEXT,
    scenario TEXT,
    target_user TEXT,
    assistant_role TEXT,
    welcome_message TEXT,
    fallback_message TEXT,
    prompt_extra TEXT,
    enabled TINYINT(1) DEFAULT 1,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    INDEX idx_business_line_enabled (enabled)
);
```

默认记录：

```text
id = "default_business_line"
```

------

#### 2.3 knowledge_base

用途：

保存知识库信息。

表名：

```text
knowledge_base
```

建表语句：

```sql
CREATE TABLE knowledge_base (
    kb_id VARCHAR(64) PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    description TEXT,
    enabled TINYINT(1) DEFAULT 1,
    doc_count INT DEFAULT 0,
    chunk_count INT DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    INDEX idx_knowledge_base_enabled (enabled)
);
```

说明：

```text
enabled = 1 的知识库可以参与用户端检索
实际检索范围还需要同时满足：该知识库已绑定当前业务线
doc_count 用于后台页面展示
chunk_count 用于后台页面展示
```

------

#### 2.4 business_line_knowledge_base

用途：

保存业务线和知识库的多对多绑定关系。

表名：

```text
business_line_knowledge_base
```

建表语句：

```sql
CREATE TABLE business_line_knowledge_base (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    business_line_id VARCHAR(64) NOT NULL,
    kb_id VARCHAR(64) NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,

    UNIQUE KEY uk_business_line_kb (business_line_id, kb_id),
    INDEX idx_blk_business_line (business_line_id),
    INDEX idx_blk_kb (kb_id)
);
```

说明：

```text
业务线编辑页维护该表。
一个业务线必须至少绑定一个启用知识库后，用户端问答才有可检索范围。
知识库禁用时绑定关系不删除，只是不参与检索。
```

#### 2.5 document

用途：

保存上传文档信息和解析状态。

表名：

```text
document
```

建表语句：

```sql
CREATE TABLE document (
    doc_id VARCHAR(64) PRIMARY KEY,
    kb_id VARCHAR(64) NOT NULL,
    filename VARCHAR(255) NOT NULL,
    file_type VARCHAR(32),
    file_path VARCHAR(1024),
    minio_url VARCHAR(1024),
    parse_status VARCHAR(32) DEFAULT 'pending',
    chunk_count INT DEFAULT 0,
    image_count INT DEFAULT 0,
    error_msg TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    INDEX idx_document_kb_id (kb_id),
    INDEX idx_document_parse_status (parse_status)
);
```

parse_status 枚举：

```text
pending
parsing
success
failed
```

#### 2.6 document_image

用途：

保存文档中提取出来的图片元数据。

表名：

```text
document_image
```

建表语句：

```sql
CREATE TABLE document_image (
    image_id VARCHAR(64) PRIMARY KEY,
    doc_id VARCHAR(64) NOT NULL,
    kb_id VARCHAR(64) NOT NULL,
    filename VARCHAR(255),
    minio_url VARCHAR(1024) NOT NULL,
    caption TEXT,
    alt_text TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    INDEX idx_document_image_doc_id (doc_id),
    INDEX idx_document_image_kb_id (kb_id)

```

说明：

```text
caption：图片识别模型生成的图片描述
alt_text：图片原始 alt 文本，如果有则保存
minio_url：前端展示图片时使用
```

#### 2.8 import_task，可选

用途：

如果文档解析后续要做异步任务，可以保存导入任务状态。第一版同步解析可以不做。

表名：

```text
import_task
```

建表语句：

```sql
CREATE TABLE import_task (
    task_id VARCHAR(64) PRIMARY KEY,
    doc_id VARCHAR(64) NOT NULL,
    kb_id VARCHAR(64) NOT NULL,
    status VARCHAR(32) DEFAULT 'pending',
    current_step VARCHAR(255),
    error_msg TEXT,
    started_at DATETIME,
    finished_at DATETIME,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    INDEX idx_import_task_doc_id (doc_id),
    INDEX idx_import_task_status (status)
);
```

状态枚举：

```text
pending
running
success
failed
```

### 3. MongoDB Collection 设计

MongoDB 用于保存聊天会话、聊天消息、问题改写结果、引用快照等非结构化或半结构化数据。

------

#### 3.1 chat_sessions

用途：

保存用户会话。

Collection：

```text
chat_sessions
```

文档结构：

```json
{
  "session_id": "session_xxx",
  "company_id": "default_company",
  "business_line_id": "business_line_course",
  "created_at": "2026-06-12T10:00:00",
  "updated_at": "2026-06-12T10:10:00",
  "message_count": 10,
  "last_message": "RAG 的基本流程是什么？",
  "subject_names": ["RAG", "Milvus", "Embedding"]
}
```

索引建议：

```javascript
db.chat_sessions.createIndex({ "session_id": 1 }, { unique: true })
db.chat_sessions.createIndex({ "business_line_id": 1, "updated_at": -1 })
db.chat_sessions.createIndex({ "updated_at": -1 })
```

说明：

```text
session_id：前端生成并保存到 cookie 或 localStorage
message_count：当前 session 下消息数量
last_message：用于后台会话列表展示
subject_names：该会话涉及的主要主体
```

------

#### 3.2 chat_messages

用途：

保存每一条聊天消息。

Collection：

```text
chat_messages
```

用户消息结构：

```json
{
  "message_id": "msg_xxx",
  "session_id": "session_xxx",
  "company_id": "default_company",
  "business_line_id": "business_line_course",
  "role": "user",
  "content": "那它和微调有什么区别？",
  "rewritten_query": "RAG 和模型微调有什么区别？",
  "subject_names": ["RAG", "模型微调"],
  "created_at": "2026-06-12T10:00:00"
}
```

助手消息结构：

```json
{
  "message_id": "msg_xxx",
  "session_id": "session_xxx",
  "company_id": "default_company",
  "business_line_id": "business_line_course",
  "role": "assistant",
  "content": "RAG 和模型微调的区别主要在于……",
  "references": [
    {
      "chunk_id": "chunk_xxx",
      "kb_id": "kb_xxx",
      "doc_id": "doc_xxx",
      "filename": "rag.md",
      "title": "RAG 基础",
      "content": "RAG 是一种检索增强生成技术……",
      "score": 0.82,
      "image_urls": [
        "http://localhost:9000/kb-images/kb_001/doc_001/img_001.png"
      ],
      "subject_names": ["RAG", "Embedding"]
    }
  ],
  "subject_names": ["RAG", "模型微调"],
  "created_at": "2026-06-12T10:00:05"
}
```

索引建议：

```javascript
db.chat_messages.createIndex({ "message_id": 1 }, { unique: true })
db.chat_messages.createIndex({ "session_id": 1, "created_at": 1 })
db.chat_messages.createIndex({ "business_line_id": 1, "created_at": -1 })
db.chat_messages.createIndex({ "created_at": -1 })
```

说明：

```text
用户消息保存原始问题、改写问题和主体识别结果
助手消息保存回答内容和引用来源快照
references 使用快照保存，避免后续文档删除后聊天记录无法展示原引用
```

### 4. MinIO 存储设计

MinIO 用于保存原始文档和文档图片。

Bucket 建议：

```text
kb-files
kb-images
```

路径规则：

```text
kb-files/{kb_id}/{doc_id}/{filename}
kb-images/{kb_id}/{doc_id}/{image_id}.png
```

示例：

```text
kb-files/kb_001/doc_001/rag.md
kb-images/kb_001/doc_001/img_001.png
```

要求：

```text
图片 URL 必须能被前端访问
原始文档可以设置私有
图片文件建议公开读，方便聊天框展示
```

### 5. Milvus Collection 设计

Milvus 用于保存知识库 chunk、向量、主体识别结果和图片 URL。

Collection 名称：

```text
kb_chunks
```

字段设计：

```text
chunk_id       VARCHAR PRIMARY KEY
kb_id          VARCHAR
doc_id         VARCHAR
filename       VARCHAR
title          VARCHAR
title_path     VARCHAR
content        VARCHAR
chunk_index    INT64
subject_names  VARCHAR
image_urls     VARCHAR
dense_vector   FLOAT_VECTOR
sparse_vector  SPARSE_FLOAT_VECTOR，可选
created_at     VARCHAR
```

字段说明：

```text
chunk_id：chunk 唯一 ID
kb_id：所属知识库 ID
doc_id：所属文档 ID
filename：来源文件名
title：当前 chunk 标题
title_path：完整标题路径
content：chunk 正文内容，必须是字符串
chunk_index：chunk 顺序
subject_names：主体识别结果，JSON 字符串
image_urls：图片 URL 列表，JSON 字符串
dense_vector：BGE-M3 dense 向量
sparse_vector：BGE-M3 sparse 向量，可选
```

subject_names 示例：

```json
["RAG", "Milvus", "Embedding"]
```

image_urls 示例：

```json
[
  "http://localhost:9000/kb-images/kb_001/doc_001/img_001.png"
]
```

查询过滤规则：

```text
用户提问时，先根据 business_line_id 查询绑定关系
只保留 knowledge_base.enabled = 1 的知识库 ID
然后在 Milvus 中使用 kb_id in [...] 过滤检索
```

删除规则：

```text
删除文档时，根据 doc_id 删除 Milvus chunk
删除知识库时，根据 kb_id 删除 Milvus chunk
```

注意事项：

```text
content 一定是 str，不能是 list
subject_names 和 image_urls 如果 Milvus 不方便存数组，统一 JSON 字符串化
sparse_vector 第一版实现成本高的话可以先不启用，只做 dense 检索
```
