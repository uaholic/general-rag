# FastAPI 框架与 LangGraph 设计草案

## 一、文档目标

本文用于确定后端框架、接口分层、LangGraph 状态与节点设计，并记录当前已经允许直接落地的基础代码。

当前阶段边界：

```text
可以直接落地：
FastAPI app 骨架、前端静态文件挂载、schema/router stub、SSE 事件协议、MySQL 基础连接框架。

暂时不做：
真实登录鉴权、统一异常处理、完整业务 CRUD、完整 LangGraph 节点实现。
```

------

## 二、整体分层原则

项目后端建议分成两类服务：

```text
import 服务：
管理端、导入侧、文档处理、后台配置、后台测试。

query 服务：
用户端聊天框配置、用户端聊天、聊天历史查询。
```

和当前目录对应关系：

```text
app/api
  FastAPI app、router、schema、依赖注入

app/process/import_/agent
  文档导入 LangGraph：state、main_graph、nodes

app/process/query/agent
  用户问答 LangGraph：state、main_graph、nodes

app/rag/import_
  导入节点调用的具体业务实现，例如解析、切分、图片处理、向量入库

app/rag/query
  查询节点调用的具体业务实现，例如改写、检索、rerank、组装 prompt、生成回答

app/infra
  外部基础设施适配层，例如 LLM、Milvus、MinIO、MongoDB、MySQL Repository

app/infra/persistence
  MySQL engine、Session、BaseRepository、后续 ORM 模型与业务 Repository

app/resources/frontend
  后台原型页面与静态资源。FastAPI 对外仍暴露 /admin、/demo、/login、/static。

app/shared
  通用工具、配置、日志、SSE、任务状态等
```

说明：

```text

routers 只负责接口入参、出参、调用 service/graph，不直接写复杂业务。
具体业务可以先 TODO，后续逐步迁移到 infra / rag / process。
```

## 七、接口与 LangGraph 的关系

不是所有接口都需要 LangGraph。

```text
普通 CRUD 接口：
企业配置、业务线配置、知识库配置、模型配置、聊天记录查询。
这些接口不需要 LangGraph，直接走 Repository / Service 即可。

导入类接口：
上传文档、重新解析文档。
这些接口触发 import graph。

查询类接口：
用户流式聊天、后台问答测试。
这些接口触发 query graph。

配置类用户接口：
/api/widget/config
只查询配置和绑定关系，不触发 graph。
```

接口到图的映射：

```text
POST /admin/documents/upload
  -> import graph

POST /admin/documents/{doc_id}/reparse
  -> import graph

POST /admin/playground/chat
  -> query graph，source=playground

POST /api/chat/stream
  -> query graph，source=widget
```

## 八、ImportGraph 设计

### 8.1 ImportGraphState 草案

建议文件：

```text
app/process/import_/agent/state.py
```

```python
from typing import Any, TypedDict


class ImportGraphState(TypedDict, total=False):
    task_id: str
    kb_id: str
    doc_id: str

    filename: str
    file_type: str
    local_file_path: str
    minio_object_name: str
    minio_url: str

    raw_text: str
    markdown_text: str
    parsed_text: str

    images: list[dict[str, Any]]
    image_urls: list[str]
    image_descriptions: list[dict[str, Any]]

    chunks: list[dict[str, Any]]
    subject_names: list[str]
    embeddings: dict[str, list]

    milvus_insert_count: int
    doc_count_delta: int
    chunk_count_delta: int

    status: str
    current_node: str
    done_nodes: list[str]
    error: str
```

字段说明：

```text
task_id：
导入任务 ID，用于后台进度和 SSE。

kb_id / doc_id：
当前文档所属知识库和文档记录。

raw_text / markdown_text / parsed_text：
不同阶段的文本结果。后续可以收敛为一个字段，但练习阶段拆开更清楚。

images / image_urls / image_descriptions：
图片抽取、上传和视觉模型描述的中间结果。

chunks：
切分后的文档块，最终写入 Milvus。

subject_names：
文档或 chunk 识别出来的主体关键词。

embeddings：
BGE-M3 生成的 dense/sparse 向量。

status / current_node / done_nodes / error：
用于后台进度展示和问题排查。
```

### 8.2 ImportGraph 节点清单

```text
node_prepare_import
  初始化任务状态，校验 kb_id、doc_id、文件路径。

node_save_original_file
  保存原始文件到 MinIO 或本地。

node_parse_document
  根据文件类型解析 Markdown / TXT / PDF。
  PDF 可以先 TODO，后续接 MinerU。

node_extract_images
  从 Markdown/PDF 解析结果中提取图片信息。

node_upload_images
  上传图片到 MinIO，生成 image_urls。

node_describe_images
  调用视觉模型生成图片说明。

node_merge_image_descriptions
  将图片 URL 和图片说明合并回 markdown_text。

node_split_document
  按标题、段落或 token 长度切分 chunk。

node_subject_recognition
  识别文档主体关键词，例如 RAG、Milvus、Embedding。

node_generate_embeddings
  调用 BGE-M3，为 chunks 生成 dense/sparse 向量。

node_write_milvus
  将 chunk、metadata、向量写入 Milvus。

node_update_document_stats
  更新 document、knowledge_base 的 parse_status、chunk_count、doc_count。

node_finish_import
  标记任务成功。

node_fail_import
  标记任务失败，写入错误信息。
```

### 8.3 ImportGraph 边设计

```text
START
  -> node_prepare_import
  -> node_save_original_file
  -> node_parse_document
  -> node_extract_images
  -> 条件判断：是否有图片
      有图片 -> node_upload_images -> node_describe_images -> node_merge_image_descriptions
      无图片 -> node_split_document
  -> node_split_document
  -> node_subject_recognition
  -> node_generate_embeddings
  -> node_write_milvus
  -> node_update_document_stats
  -> node_finish_import
  -> END
```

异常处理建议：

```text
第一版可以在 graph 外层 try/except 捕获异常，统一更新 import_task 和 document.parse_status。
后续再为每个节点做更细的失败边。
```

### 8.4 ImportGraph 节点与 rag/import_ 的职责边界

```text
nodes 里只做：
1. 从 state 取参数
2. 调用 rag/import_ 的业务函数
3. 将结果写回 state
4. 更新任务进度

rag/import_ 里做：
1. 文档解析
2. 图片处理
3. chunk 切分
4. 主体识别
5. embedding
6. Milvus 入库数据组装
```

## 九、QueryGraph 设计

### 9.1 QueryGraphState 草案

建议文件：

```text
app/process/query/agent/state.py
```

```python
from typing import Any, TypedDict


class QueryGraphState(TypedDict, total=False):
    source: str
    session_id: str
    company_id: str
    business_line_id: str

    question: str
    rewritten_query: str
    history_messages: list[dict[str, Any]]

    company_config: dict[str, Any]
    business_line_config: dict[str, Any]
    model_config: dict[str, Any]
    bound_kb_ids: list[str]

    user_message_id: str

    query_embedding: dict[str, Any]
    milvus_expr: str
    retrieved_chunks: list[dict[str, Any]]
    reranked_chunks: list[dict[str, Any]]

    prompt: str
    answer: str
    answer_delta: str
    references: list[dict[str, Any]]

    fallback_message: str
    should_answer: bool
    error: str
```

字段说明：

```text
source：
widget 或 playground。后台测试和用户端聊天可以复用同一套 query graph。

business_line_id：
必须存在，决定检索范围和助手身份。

bound_kb_ids：
从 MySQL 绑定表查询得到，并过滤 knowledge_base.enabled = 1。

milvus_expr：
例如 kb_id in ["kb_course", "kb_milvus"]。

retrieved_chunks / reranked_chunks：
检索结果和重排结果分开保存，便于后台测试展示。

references：
最终给前端展示、也会保存到 MongoDB assistant message 中。
```

### 9.2 QueryGraph 节点清单

```text
node_validate_query
  校验 session_id、business_line_id、question。

node_load_runtime_config
  读取 company_config、business_line_config、model_config。

node_load_bound_knowledge_bases
  查询当前业务线绑定且启用的知识库 ID。
  如果为空，走 fallback。

node_save_user_message
  保存用户原始问题到 MongoDB。

node_load_history
  查询最近 N 轮聊天历史。

node_rewrite_query
  调用 LLM 结合历史改写用户问题。

node_generate_query_embedding
  调用 BGE-M3 生成查询向量。

node_search_milvus
  使用 bound_kb_ids 生成过滤表达式，执行向量检索。

node_rerank_chunks
  可选节点，根据 model_config.use_rerank 决定是否执行。

node_check_retrieval_empty
  如果没有可用 chunk，设置 should_answer=False。

node_build_answer_prompt
  组装企业信息、业务线信息、历史、检索资料和回答约束。

node_generate_answer
  调用 LLM 生成回答。
  widget 场景需要流式输出；playground 可以先非流式。

node_build_references
  从最终 chunk 中提取引用来源、图片 URL、score、标题等。

node_save_assistant_message
  保存助手回答和 references 到 MongoDB。

node_finish_query
  返回最终 answer、references、rewritten_query。
```

### 9.3 QueryGraph 边设计

```text
START
  -> node_validate_query
  -> node_load_runtime_config
  -> node_load_bound_knowledge_bases
  -> 条件判断：bound_kb_ids 是否为空
      为空 -> node_build_fallback_answer -> node_save_assistant_message -> END
      不为空 -> node_save_user_message
  -> node_load_history
  -> node_rewrite_query
  -> node_generate_query_embedding
  -> node_search_milvus
  -> 条件判断：是否启用 rerank
      启用 -> node_rerank_chunks
      不启用 -> node_check_retrieval_empty
  -> node_check_retrieval_empty
  -> 条件判断：是否有可回答依据
      无依据 -> node_build_fallback_answer -> node_save_assistant_message -> END
      有依据 -> node_build_answer_prompt
  -> node_generate_answer
  -> node_build_references
  -> node_save_assistant_message
  -> node_finish_query
  -> END
```

### 9.4 QueryGraph 与 SSE 的关系

用户端 `/api/chat/stream` 需要 SSE。

建议事件：

```text
ready：
连接建立。

progress：
节点进度，例如正在改写、正在检索、正在生成回答。

建议 data：

```json
{
  "step": "retrieve",
  "label": "检索相关资料",
  "current": 3,
  "total": 5,
  "percent": 60,
  "message": "正在检索当前业务线绑定的知识库"
}
```

rewrite：
返回 rewritten_query。

delta：
LLM 流式输出片段。

image：
回答过程中或回答完成时展示图片。

建议 data：

```json
{
  "placement": "inline",
  "message": "下面是资料中提到的流程图",
  "images": [
    {
      "image_id": "img_001",
      "url": "http://127.0.0.1:9000/enterprise-rag/kb-images/xxx.png",
      "caption": "RAG 流程图",
      "doc_id": "doc_001",
      "page_no": 3
    }
  ]
}
```

说明：

```text
不建议把事件命名为 showimg。
原因是 showimg 暗含一种固定 UI 行为。
使用 image 事件更通用，前端可以根据 placement 决定内联展示、引用区展示，或后续点击放大。
```

references：
回答完成后返回引用来源。

final：
返回完整 answer、references、message_id。

error：
异常信息。
```

第一版可简化：

```text
ready -> progress 多次 -> delta 多次 -> image 可选 -> references -> final
```
