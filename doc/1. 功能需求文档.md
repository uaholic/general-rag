# 轻量级通用企业知识库系统需求文档

## 一、功能需求

### 1. 项目定位

本项目是一个面向企业业务线的轻量级 RAG 知识库问答系统。

系统允许管理员配置企业信息、业务线信息、知识库和模型预设，上传包含文字和图片的业务文档。系统会自动完成文档解析、图片识别、chunk 切分、主体识别、embedding、Milvus 入库，并提供一个可以嵌入企业网站的 JS 聊天框。

终端用户通过聊天框提问时，系统根据当前业务线绑定且启用的知识库进行检索，结合聊天历史进行问题改写，调用大模型生成流式回答，并展示引用来源、相关片段和图片。

第一版系统默认：

```text
单企业
多业务线
每个业务线一个聊天框嵌入代码
多知识库
业务线可绑定一个或多个知识库
多文档
固定模型预设
```

第一版不做：

```text
多租户
真实账号权限
复杂机器人管理
自由节点编排
Agent 工作流
计费系统
人工客服接管
复杂数据分析报表
```

------

### 2. 系统核心目标

系统最终应支持：

```text
1. 管理员通过暗号登录后台。
2. 管理员配置企业基础信息。
3. 管理员创建、编辑、启用 / 禁用业务线。
4. 管理员创建和管理知识库。
5. 管理员为每个业务线绑定一个或多个知识库。
6. 管理员在业务线列表中查看并复制该业务线的嵌入代码。
7. 管理员上传 Markdown、TXT、PDF 文档。
8. 系统解析文档正文内容。
9. 系统提取文档中的图片。
10. 系统将图片上传到 MinIO。
11. 系统调用图片识别模型生成图片描述。
12. 系统将图片 Markdown 和图片说明写入文档内容。
13. 系统对文档内容进行 chunk 切分。
14. 系统对 chunk 进行主体识别。
15. 系统调用 BGE-M3 生成 embedding。
16. 系统将 chunk 和向量写入 Milvus。
17. 用户端通过 JS 聊天框提问。
18. 系统根据 session/cookie 识别用户会话。
19. 系统保存聊天记录到 MongoDB。
20. 系统根据聊天历史进行问题改写。
21. 系统只检索当前业务线绑定且启用的知识库中的相关 chunk。
22. 系统调用 LLM 生成流式回答。
23. 聊天框支持 Markdown 渲染。
24. 聊天框支持图片展示。
25. 聊天框展示引用来源。
26. 后台可以查看和清空聊天记录。
27. 后台展示模型预设配置。
```

#### 4.5 文档上传与解析

路径：

```text
/admin/documents
```

功能：

```text
上传文档
选择所属知识库
查看文档解析状态
查看解析后的 chunk 数量
查看解析错误信息
重新解析
删除文档
```

支持文件类型：

```text
Markdown
TXT
PDF
```

优先级：

```text
P0：Markdown、TXT
P1：PDF
```

文档状态：

```text
pending    待解析
parsing    解析中
success    解析成功
failed     解析失败
```

文档处理流程：

```text
上传文档
  ↓
保存原始文件到 MinIO 或本地
  ↓
写入 MySQL document 记录，状态 pending
  ↓
更新状态为 parsing
  ↓
解析正文文本
  ↓
提取文档中的图片
  ↓
图片上传 MinIO
  ↓
调用图片识别模型生成图片描述
  ↓
将图片链接和图片描述插入 Markdown 内容
  ↓
chunk 切分
  ↓
主体识别
  ↓
embedding
  ↓
Milvus 入库
  ↓
更新文档状态、chunk 数量、图片数量
  ↓
更新知识库 doc_count、chunk_count
```

失败处理：

```text
任意步骤失败
  ↓
文档状态更新为 failed
  ↓
记录 error_msg
```

#### 4.7 主体识别

主体识别用于从 chunk 中抽取关键业务主体、技术名词、产品名、概念名，方便后续检索、过滤、聊天历史分析和推荐问题。

示例主体：

```text
RAG
Milvus
LangChain
LangGraph
Embedding
Rerank
BGE-M3
SSE
MinIO
MongoDB
```

主体识别结果保存字段：

```text
subject_names
```

第一版实现方式：

```text
规则抽取 + LLM 抽取二选一
```

推荐第一版先用 LLM 抽取，每个 chunk 抽取 3 到 8 个主体。

示例 Prompt：

```text
请从下面文本中抽取关键主体、技术名词或业务名词。
要求：
1. 返回 JSON 数组。
2. 每个主体不要超过 20 个字。
3. 不要解释。
4. 最多返回 8 个。

文本：
{chunk_content}
```

示例返回：

```json
["RAG", "Milvus", "Embedding", "Rerank", "向量检索"]
```

用途：

```text
存入 Milvus chunk 字段
聊天历史中记录关联主体
后续支持主体过滤
后续支持推荐问题
后续支持会话主题分析
```

第一版性能降级策略：

```text
如果每个 chunk 都调 LLM 太慢，可以先只用规则抽取。
也可以只对文档标题和前若干个 chunk 做 LLM 抽取。
```

#### 5.3 历史上下文问题改写

功能说明：

当用户连续提问时，后端可以根据最近几轮聊天历史，将用户当前问题改写成更完整的问题。

示例：

历史对话：

```text
用户：RAG 是什么？
助手：RAG 是检索增强生成……
用户：那它和微调有什么区别？
```

改写后：

```text
RAG 和模型微调有什么区别？
```

流程：

```text
接收用户问题
  ↓
根据 session_id 查询最近 N 轮聊天历史
  ↓
调用 LLM 进行问题改写
  ↓
用改写后的问题进行 embedding 和检索
  ↓
回答时仍展示用户原始问题
```

第一版规则：

```text
默认取最近 3 轮对话
问题改写失败时使用原始问题
改写后的问题保存到 MongoDB chat_messages.rewritten_query
```

### 6. RAG 问答流程

完整问答流程：

```text
用户打开 demo.html
  ↓
chat-widget.js 加载
  ↓
生成或读取 session_id
  ↓
请求 /api/widget/config 获取欢迎语
  ↓
用户输入问题
  ↓
请求 /api/chat/stream
  ↓
后端保存 user message 到 MongoDB
  ↓
读取最近 N 轮聊天历史
  ↓
LLM 改写用户问题
  ↓
读取企业和业务线配置
  ↓
读取模型配置
  ↓
读取当前业务线绑定且启用的知识库
  ↓
改写问题 embedding
  ↓
Milvus 按 kb_id 过滤检索
  ↓
rerank，可选
  ↓
构造 prompt
  ↓
调用 LLM 流式输出
  ↓
前端展示回答
  ↓
返回 references
  ↓
保存 assistant message 到 MongoDB
```

Prompt 模板：

```text
你是 {company_name} 的 {business_line_name} 智能助手。

公司信息：
{company_description}

业务线信息：
{business_line_description}

你的回答角色：
{assistant_role}

回答要求：
1. 优先根据知识库资料回答。
2. 如果知识库没有明确依据，不要编造。
3. 回答语气：{answer_tone}
4. 如果知识库内容中包含图片链接，可以在回答中使用 Markdown 图片语法展示图片。
5. 如果涉及风险、费用、投资、贷款、保险、法律、医疗等内容，需要保持谨慎。
6. 必要时附加提示：{disclaimer}
7. 如果无法回答，使用以下话术：{fallback_message}

历史对话：
{history}

知识库资料：
{retrieved_chunks}

用户原始问题：
{original_question}

改写后的检索问题：
{rewritten_query}
```
